# Cole Sebeol Keyboard Specification
#
# Provenance
#   The key rows in this file are an independent transcription of the Gong
#   Byung-woo Sebeolsik Final layout into Hisle's own specification format.
#   The libhangul project was consulted for Unicode Hangul values, Sebeolsik
#   layout data, and behavior comparison, but this file does not vendor
#   libhangul XML data. See the repository-level NOTICE for attribution notes.
#
# Purpose
#   Reference data for Hisle's Cole Sebeol layout.
#   Cole Sebeol uses the Gong Byung-woo Sebeolsik Final key map on standard US
#   QWERTY representative key positions. Colemak is the underlying Roman layout
#   for shortcut forwarding and Roman-mode behavior; it is not used to derive
#   Hangul jamo assignments.
#
# Cole Sebeol policy
#   - Hangul jamo assignments are derived from the Sebeolsik Final layout on
#     representative QWERTY key positions.
#   - This specification names physical key positions with 대표 글쇠
#     (representative keys): standard US QWERTY key labels. Representative keys
#     are intentionally separate from Colemak underlying Roman letters. For
#     example, representative keys j k f name physical J, K, F; if written as
#     Colemak underlying letters, the same sequence would be n e t.
#   - The underlying Roman layout is Colemak, not QWERTY. The complete
#     representative-QWERTY to Colemak mapping is listed in underlying-key rows
#     below. This mapping is used for shortcut forwarding and Roman-mode text,
#     not for Hangul jamo assignment.
#   - "Underlying Roman layout" means modifier shortcuts are interpreted and
#     forwarded as Colemak Roman shortcuts while Hangul input is active. Host
#     applications receive Command/Control chords such as Command+C or
#     Command+V as Roman shortcut events, never as Hangul jamo. For letters that
#     differ between QWERTY and Colemak, the Colemak letter is authoritative.
#     For example, the physical key that QWERTY labels E is Colemak F, so
#     Command plus that key should be delivered as Command+F.
#   - Hangul jamo assignments and policy examples are written against
#     representative keys. Modifier shortcut behavior still follows the
#     Colemak underlying Roman layout.
#
# sane-punctuation policy
#   - This policy is named sane-punctuation.
#   - Cole Sebeol Hangul mode keeps Hangul Jamo key rows as composition input,
#     but printable non-jamo output follows the sane-punctuation table
#     below instead of the Sebeolsik Final symbol layer.
#   - sane-punctuation is separate from the underlying Roman layout. The
#     underlying Roman layout still controls shortcut forwarding and Roman-mode
#     text. sane-punctuation controls Hangul-mode printable scalar output.
#   - A non-jamo key row still uses FlushThenEmit: first flush any active
#     composition, then emit the printable scalar from the key row.
#   - The policy table is written in representative-key notation. The last
#     row's middle key is representative n; its Colemak underlying key is k.
#     If a UI labels keys by the underlying Roman layout, that same physical
#     row may appear as b k m , . /.
#   - The sane-punctuation table is:
#       representative keys:
#         `1234567890-=
#         []\
#         ;'
#         b n m , . /
#       unshifted output:
#         `ㅎㅆㅂㅛㅠㅑㅖㅢㅜㅋ-=
#         []\
#         ㅂㅌ
#         ㅜ ㅅ ㅎ , . ㅗ
#       shifted output:
#         ~ㄲㄺㅈㄿㄾ^&*()_+
#         {}|
#         4"
#         ? : ; < > !
#
# 초성 선행 모아치기 정책
#   - This policy is named 초성 선행 모아치기 정책.
#   - Cole Sebeol does not support full 모아치기.
#   - A syllable must resolve its choseong first. A jungseong or jongseong typed
#     before the choseong is not held for a later choseong and must not be
#     retroactively folded into that later syllable.
#   - After the choseong is resolved, the syllable may accept its jungseong and
#     jongseong in either order. For example, choseong-jungseong-jongseong and
#     choseong-jongseong-jungseong are equivalent when the same jamo are used.
#   - This is the only supported 모아치기 behavior: the engine may reorder a
#     post-choseong jungseong and jongseong inside the active syllable, but it
#     must not reorder a choseong behind an already typed jungseong or jongseong.
#   - If a second choseong arrives while a choseong-only syllable is active, the
#     existing choseong may combine through the table below when a choseong
#     combination exists. Otherwise, the previous choseong is flushed and the
#     new choseong starts the next syllable.
#   - The choseong-combination state closes deterministically as soon as a
#     jungseong or jongseong arrives. A later choseong must not be combined with
#     a choseong that is already followed by a jungseong or jongseong.
#   - If another choseong arrives after the active syllable has a jungseong or
#     jongseong, the active syllable is flushed and the new choseong starts the
#     next syllable.
#   - Test coverage for this policy must include both accepted and rejected
#     orderings:
#       k f x -> 각
#       k x f -> 각
#       x k f -> ㄱ가
#       f k x -> ㅏㄱㄱ
#
# slash-nine(/9) 정책
#   - This policy is named slash-nine(/9) 정책.
#   - slash-nine(/9) 정책 means the representative / key's ㅗ and the
#     representative 9 key's ㅜ are source-key-special vowels. The name refers
#     to the / and 9 representative key positions, not to the v and b
#     representative key positions that carry the same Unicode jamo scalars.
#   - This policy exists for crying expressions that use repeated ㅜ. For
#     example, 망했어ㅜㅜ must stay 망했어ㅜㅜ; typing representative b-key ㅜ
#     after the ㅓ in 어 must not fold that ㅓ+ㅜ into ㅝ and produce 망했워ㅜ.
#   - The two representative ㅗ keys and two representative ㅜ keys share Unicode
#     jamo scalars, but they do not have identical composition behavior. The
#     automata must retain the source key for U+1169 and U+116E instead of
#     reducing input to a jamo scalar only.
#   - The representative / key's U+1169 and representative 9 key's U+116E may
#     appear either before or after the matching base jungseong:
#       / f, f / -> U+116A  # ㅗ + ㅏ, ㅏ + ㅗ -> ㅘ
#       / r, r / -> U+116B  # ㅗ + ㅐ, ㅐ + ㅗ -> ㅙ
#       / d, d / -> U+116C  # ㅗ + ㅣ, ㅣ + ㅗ -> ㅚ
#       9 t, t 9 -> U+116F  # ㅜ + ㅓ, ㅓ + ㅜ -> ㅝ
#       9 c, c 9 -> U+1170  # ㅜ + ㅔ, ㅔ + ㅜ -> ㅞ
#       9 d, d 9 -> U+1171  # ㅜ + ㅣ, ㅣ + ㅜ -> ㅟ
#   - The representative v key's U+1169 and representative b key's U+116E may
#     produce compound vowels only when they are the first source key:
#       v f -> U+116A  # ㅗ + ㅏ -> ㅘ
#       v r -> U+116B  # ㅗ + ㅐ -> ㅙ
#       v d -> U+116C  # ㅗ + ㅣ -> ㅚ
#       b t -> U+116F  # ㅜ + ㅓ -> ㅝ
#       b c -> U+1170  # ㅜ + ㅔ -> ㅞ
#       b d -> U+1171  # ㅜ + ㅣ -> ㅟ
#   - The reverse representative source-key orders f v, r v, d v, t b, c b,
#     and d b must not combine. They remain standalone vowel sequences ㅏㅗ,
#     ㅐㅗ, ㅣㅗ, ㅓㅜ, ㅔㅜ, and ㅣㅜ.
#   - The source-combine rows below are authoritative for all slash-nine(/9)
#     and v/b compound vowels. Do not add generic scalar-only combine rows such
#     as U+1169+U+1175 or U+116E+U+1175 for these source-sensitive vowels; those
#     rows would make d v and d b combine incorrectly.
#   - Test coverage for this policy must include both accepted and rejected
#     source-key orderings:
#       / f, f /, v f -> ㅘ; f v -> ㅏㅗ
#       / r, r /, v r -> ㅙ; r v -> ㅐㅗ
#       / d, d /, v d -> ㅚ; d v -> ㅣㅗ
#       9 t, t 9, b t -> ㅝ; t b -> ㅓㅜ
#       9 c, c 9, b c -> ㅞ; c b -> ㅔㅜ
#       9 d, d 9, b d -> ㅟ; d b -> ㅣㅜ
#
# 약한 아래 정책
#   - This policy is named 약한 아래 정책.
#   - 약한 아래 정책 keeps the Sebeolsik Final-derived layout while taking part
#     of 순아래's no-shift advantage. It is intentionally weaker than 순아래:
#     only the explicit pairs in this section are added.
#   - 약한 아래 정책 does not logically require 초성 선행 모아치기 정책, but
#     초성 선행 모아치기 정책 provides the boundary that makes 약한 아래 stable
#     in this sequential automata. In particular, weak-below choseong pairs are
#     applied while resolving the initial consonant position, not by reordering a
#     late choseong behind already typed jungseong or jongseong.
#   - Therefore, j k f becomes 까, but j f k becomes 아ㄱ. The f jungseong closes
#     the choseong-combination state after j, so the later k cannot combine with
#     that j to make ㄲ.
#   - The order of each pair does not matter. For example, choseong ieung plus
#     choseong kiyeok and choseong kiyeok plus choseong ieung both produce
#     choseong ssangkiyeok.
#   - Same-jamo choseong repetition does not trigger these tense consonants. In
#     particular, 초성 ㄱ+ㄱ, ㅈ+ㅈ, ㅂ+ㅂ, ㄷ+ㄷ, and ㅅ+ㅅ must stay as two
#     adjacent jamo positions instead of becoming 초성 ㄲ, ㅉ, ㅃ, ㄸ, or ㅆ.
#     Jongseong same-jamo combinations such as 종성 ㄱ+ㄱ -> 종성 ㄲ and
#     종성 ㅅ+ㅅ -> 종성 ㅆ are still allowed.
#   - The 약한 아래 pairs are:
#       choseong ieung + choseong kiyeok -> choseong ssangkiyeok
#       choseong kiyeok + choseong cieuc -> choseong ssangcieuc
#       choseong cieuc + choseong pieup -> choseong ssangpieup
#       choseong tikeut + choseong mieum -> choseong ssangtikeut
#       choseong sios + choseong hieuh -> choseong ssangsios
#       jungseong i + jungseong a -> jungseong yae
#       jungseong a + jungseong i -> jungseong yae
#       jungseong i + jungseong eu -> jungseong yi
#       jungseong yeo + jungseong ae -> jungseong ye
#       jungseong ae + jungseong yeo -> jungseong ye
#       jongseong ieung + jongseong nieun -> jongseong nieun-hieuh
#       jongseong sios + jongseong rieul -> jongseong pieup-sios
#       jongseong pieup + jongseong ssangsios -> jongseong cieuc
#       jongseong hieuh + jongseong ssangsios -> jongseong ssangkiyeok
#   - Test coverage for this policy must include both directions for each pair
#     and rejected same-jamo repetitions:
#       j k f -> 까
#       k j f -> 까
#       j f k -> 아ㄱ
#       k k f -> ㄱ가
#       k l f -> 짜
#       l k f -> 짜
#       l l f -> ㅈ자
#       l ; f -> 빠
#       ; l f -> 빠
#       ; ; f -> ㅂ바
#       u i f -> 따
#       i u f -> 따
#       u u f -> ㄷ다
#       n m f -> 싸
#       m n f -> 싸
#       n n f -> ㅅ사
#       d f -> ㅒ
#       f d -> ㅒ
#       d g -> ㅢ
#       g d -> ㅢ
#       k d f -> 걔
#       k f d -> 걔
#       k d g -> 긔
#       k g d -> 긔
#       k e r -> 계
#       k r e -> 계
#       k f a s -> 갆
#       k f s a -> 갆
#       k f q w -> 값
#       k f w q -> 값
#       k f 3 2 -> 갖
#       k f 2 3 -> 갖
#       k f 1 2 -> 갂
#       k f 2 1 -> 갂
#
# Standalone jamo flush policy
#   - The automata uses Unicode Hangul Jamo scalars internally for composition.
#   - When an incomplete syllable must be flushed, visible standalone jamo are
#     emitted as Hangul Compatibility Jamo. This produces user-facing text such
#     as 아ㄱ instead of exposing conjoining jamo scalars.
#   - Compatibility conversion is defined by compat-jamo rows below. Implementers
#     must use those rows for standalone choseong, jungseong, and jongseong.
#   - Incomplete jamo are flushed in input order.
#   - Examples after an explicit Flush or an implicit flush boundary:
#       j f k -> 아ㄱ
#       x k f -> ㄱ가
#       f k x -> ㅏㄱㄱ
#       k x . -> ㄱㄱ.
#
# Second jungseong/jongseong and no-choseong policy
#   - If a new jungseong arrives while the active composition already has a
#     jungseong, the automata first tries the slot-scoped jungseong combination
#     rules, including positional-combine and source-combine rows.
#   - If that new jungseong cannot combine with the active jungseong, flush the
#     previous active composition and start a new no-choseong composition with
#     the new jungseong.
#   - If a new jungseong arrives while the active composition already has a
#     choseong, jungseong, and jongseong, flush the previous active syllable and
#     start a new no-choseong composition with the new jungseong. The previous
#     jongseong is not stolen as the choseong of the new syllable.
#   - If a new jongseong arrives while the active composition already has a
#     jongseong, the automata first tries the slot-scoped jongseong combination
#     rules.
#   - If that new jongseong cannot combine with the active jongseong, flush the
#     previous active composition and start a new no-choseong composition with
#     the new jongseong.
#   - A no-choseong composition may contain a jungseong-only stack or a
#     jongseong-only stack. Same-slot combinations are allowed when the normal
#     slot-scoped rules allow them, but the result remains no-choseong and must
#     flush as Compatibility Jamo.
#   - markedText for a no-choseong composition uses the same Compatibility Jamo
#     conversion as Flush.
#   - A no-choseong composition must not retroactively attach to a later
#     choseong. A later choseong is a partial composition boundary that flushes
#     the no-choseong composition first.
#   - A no-choseong composition cannot mix jungseong and jongseong into a
#     syllable. If a different slot arrives, flush the previous no-choseong
#     composition and start a new no-choseong composition with the new jamo.
#   - Test coverage for this policy must include:
#       k f r Flush -> 가ㅐ
#       k f x z Flush -> 각ㅁ
#       k x q f Flush -> 갃
#       k x z Flush -> ㄱㄱㅁ
#       / f Flush -> ㅘ
#       f / Flush -> ㅘ
#       x q Flush -> ㄳ
#       f r Flush -> ㅏㅐ
#       f x Flush -> ㅏㄱ
#       x f Flush -> ㄱㅏ
#       / f k f Flush -> ㅘ가
#       f / k f Flush -> ㅘ가
#       k f x f Flush -> 각ㅏ
#       k f x / Flush -> 각ㅗ
#       k x f / Flush -> 각ㅗ
#
# Automata input/output and test policy
#   - Policy examples such as k f x -> 각 mean: process the listed
#     representative printable keys in order, then explicitly Flush any active
#     composition at the end of the test.
#   - The core automata output model is fixed as:
#       committedText: newly visible text emitted by the processed input.
#       markedText: the active composition's current visible form after the
#         processed input, or the empty string when no composition is active.
#       forwardedActions: host actions that must be forwarded after any required
#         flush, such as system Backspace, system Delete, or a command/control
#         shortcut.
#   - Test fixtures should represent forwardedActions with explicit values:
#       ForwardDelete
#       ForwardBackspace
#       ForwardShortcut(modifiers: [Command|Control|Option|Shift...],
#         key: <underlying ASCII scalar>)
#     Shortcut test input is written as Command(<representative-key>) or
#     Control(<representative-key>). The forwarded shortcut key is the Colemak
#     underlying scalar from the underlying-key rows. For example:
#       k f Command(e) -> committedText "가",
#         forwardedActions [ForwardShortcut([Command], "f")]
#       Command(n) -> forwardedActions [ForwardShortcut([Command], "k")]
#   - Tests may accumulate committedText across inputs as visible text. The
#     engine stores markedText separately; markedText is not counted as committed
#     visible text until Flush or a FlushThen* boundary emits it.
#   - End-to-end examples in this document assert the final visible text after
#     the trailing explicit Flush. Boundary examples may also assert
#     forwardedActions.
#   - Printable representative keys update the active composition or flushed text
#     according to the composition policies in this document.
#   - Flush emits the active composition as visible text and clears the
#     composition state. Flush emits no extra scalar.
#   - FlushThenEmit means Flush first, then emit a printable scalar such as
#     whitespace or a non-jamo scalar from the key rows. Whitespace and key rows
#     whose values are not Hangul Jamo use FlushThenEmit. Under
#     sane-punctuation, non-jamo key rows contain the intended Hangul-mode
#     printable scalar, not the Sebeolsik Final symbol-layer scalar.
#     In the output model, committedText contains the flushed composition
#     followed by the printable scalar in that order.
#   - FlushThenForward means Flush first, then forward a non-text host action.
#     Command/control shortcuts use FlushThenForward according to the Colemak
#     underlying Roman layout. In the output model, committedText contains the
#     flushed composition and forwardedActions contains the host action.
#   - Clear clears any active composition without flushing it. Clear is reserved
#     for rare internal recovery or explicit tests; normal user paths should use
#     Flush or a FlushThen* boundary.
#   - Delete means forward delete. If composition is active, use
#     FlushThenForward(Delete). If no composition is active, let the host system
#     perform Delete without core text output.
#   - Backspace means delete backward. If composition is active, remove the last
#     representative printable key from the active composition stack and
#     recompute the active marked text. If no composition is active, let the host
#     system perform Backspace without core text output.
#   - Backspace never removes text that the automata has already flushed. For
#     example, after k f k, the second k is the active composition and 가 is
#     already flushed; Backspace removes only the active k.
#   - Incremental markedText tests should include:
#       k -> committedText "", markedText "ㄱ"
#       k x -> committedText "", markedText "ㄱㄱ"
#       k f -> committedText "", markedText "가"
#       k x f -> committedText "", markedText "각"
#       / f -> committedText "", markedText "ㅘ"
#       f / -> committedText "", markedText "ㅘ"
#       k f x f -> committedText "각", markedText "ㅏ"
#   - Test coverage for editing boundaries should include:
#       k f Backspace Flush -> ㄱ
#       k x Backspace f Flush -> 가
#       k f Delete -> 가 + ForwardDelete
#       k x Delete -> ㄱㄱ + ForwardDelete
#       Delete -> ForwardDelete
#       Backspace -> ForwardBackspace
#
# Partial composition boundary policy
#   - A pending choseong-jongseong sequence may wait only for a jungseong that
#     would complete the active syllable under 초성 선행 모아치기 정책.
#   - If a non-jungseong flush boundary arrives while only choseong and
#     jongseong are pending, the pending jamo are flushed as standalone
#     compatibility jamo in input order, and then the boundary input is
#     processed.
#   - Boundary inputs include a new choseong, non-jamo printable output,
#     whitespace, explicit Flush, Delete, and command/control shortcuts. Clear
#     is a clearing boundary: it clears the pending jamo without flushing them.
#   - Examples:
#       k x f -> 각
#       k x k -> ㄱㄱㄱ
#       k x . -> ㄱㄱ.
#       k x Space -> ㄱㄱ Space
#
# Important compatibility note:
#   In the original Sebeolsik Final behavior, left-side ㅗ and ㅜ should not
#   combine with other vowels. Hisle's Cole Sebeol automata intentionally uses
#   source-key-specific rows: / and 9 may combine on either side of their base
#   vowels, while v and b combine only when they come first.
#   약한 아래 정책 intentionally omits same-jamo choseong tense combinations,
#   and some order-insensitive weak-below pairs replace default final-consonant
#   combinations.
#
# Format
#   keyboard <id> <name> type=<type> map=<map-id> combination=<combination-id>
#   underlying-key <keyboard-id> <representative-ascii-code> <underlying-ascii-code>
#   key <keyboard-id> <representative-ascii-code> <unicode-scalar> # <key-label> -> <output>
#   compat-jamo <hangul-jamo-scalar> <compatibility-jamo-scalar>
#   combine <combination-id> <first> <second> <result>
#   positional-combine <combination-id> <first> <representative-ascii-code> <second> <result>
#   source-combine <combination-id> <first-source-key> <second-source-key> <result>
#
# Notes
#   - representative-ascii-code is the ASCII scalar for the representative key
#     label. It identifies a standard US QWERTY physical key position, not the
#     Colemak post-layout Roman scalar.
#   - underlying-key rows define the full standard US QWERTY representative key
#     to Colemak scalar mapping used for command/control shortcut forwarding.
#     Printable Hangul-mode key rows do not use this mapping.
#   - Uppercase letters and shifted punctuation are represented by their ASCII
#     scalars.
#   - Hangul values use Unicode Hangul Jamo scalars:
#       U+1100..U+115F choseong, U+1160..U+11A7 jungseong,
#       U+11A8..U+11FF jongseong.
#   - Non-jamo key values are printable scalars from the sane-punctuation table.
#     They use FlushThenEmit.
#   - compat-jamo rows define standalone flush output for internal Unicode
#     Hangul Jamo. When a composed standalone jamo such as U+116A or U+11AA
#     flushes, use its compat-jamo row.
#   - Policy examples such as k f x, j k f, v f, and b t use
#     representative-key source labels. They describe the intended key-position
#     order in the automata policy, not Colemak letters and not a global scalar
#     rewrite rule.
#   - combine rows are slot-scoped. They are not global scalar rewrites.
#     Choseong-result rows apply only while resolving the choseong position,
#     jungseong-result rows apply only while resolving the jungseong position,
#     and jongseong-result rows apply only while resolving the jongseong
#     position.
#   - positional-combine applies only when <second> comes from
#     <representative-ascii-code>.
#     Generic jamo-scalar equality is not enough for these cases.
#   - source-combine applies only to the exact two representative-key source
#     labels listed. It is used when the source key identity matters for both
#     sides.
#   - For slash-nine(/9) and v/b compound vowels, source-combine rows are the
#     complete contract. Absence of a source-combine row is intentional and means
#     the same Unicode jamo scalars must remain separate for that source-key
#     order.
